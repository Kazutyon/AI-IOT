1. CGI / FastCGI

Webサーバー（Apache/Nginx）は、静的ファイルは自分で返せるが、動的ページ（商品一覧、ログイン処理など）はアプリケーションに処理を渡す必要がある。その「アプリケーションの呼び出し方式」がCGI / FastCGIである。以前はCGIが使われていたが、今はFastCGIが標準。

Webサーバーとアプリをつなぐ仕組み

WebサーバーはHTMLや画像などの静的ファイルしか返せない。しかし現代のWebでは、ログイン処理、DBから商品一覧を取得、注文処理、JSONの生成（API）など動的処理が必要。これらはWebサーバーでは実行できないため、アプリケーションに処理を渡す仕組みが必要になる。

CGI（Common Gateway Interface）

CGIは、リクエストごとにアプリケーションを新しく起動して実行する方式。

項目内容
動作イメージWebサーバーがリクエストを受け取る → アプリケーションを新しく起動 → 入力（クエリ文字列、POSTデータ）を渡す → アプリがHTML/JSONを生成 → Webサーバーがその結果を返す
特徴仕組みが単純でわかりやすい
欠点リクエストごとにプロセスを起動するため非常に遅い。現代のWebのアクセス量には耐えられない

FastCGI（常駐方式）

FastCGIは、アプリを常駐プロセスとして起動し、Webサーバーはリクエストを渡すだけ。プロセス起動のオーバーヘッドがゼロに近い。CGIの欠点（毎回起動）を完全に解消した方式。

2. リバースプロキシ

リバースプロキシとは、ApacheやNginxがアプリケーションの「代理」としてリクエストを受け取り、内部のアプリケーションへ転送する仕組み。

なぜリバースプロキシが必要なのか

PHPならApache + php-fpm（FastCGI）で動くが、Node.js / Python / RubyはApacheだけでは動かない。理由：Apacheはアプリケーションを直接実行できない、アプリは3000番など別ポートで動く、Apacheが「入口」になってアプリへ転送する必要がある。

Apacheのリバースプロキシ設定例

Node.jsやPythonアプリをApacheが受ける例：example.comへのアクセスはApacheが受け取り、内部のlocalhost:3000（アプリ）に転送する。

リバースプロキシのメリット

  • HTTPS（SSL）をApache側で処理できる：アプリ側でSSLを扱う必要がなくなる。証明書管理が簡単、アプリはHTTPのままでよい。
  • 静的ファイルをApacheが高速に返せる：アプリに負荷をかけずに画像、CSS、JavaScriptを返せる。
  • ログ・アクセス制御をApacheに集約できる：access.log、error.log、IP制限、Basic認証、リダイレクトをApache側で統一管理できる。
  • アプリを外部に直接公開しなくてよい（安全）：アプリを3000番で直接公開すると危険。Apacheが前段に立つことで安全性が高まる。

FastCGIとリバースプロキシの違い

方式主に使う言語仕組み
FastCGI（php-fpm）PHPApache → php-fpm（FastCGI）
リバースプロキシNode.js / Python / RubyApache → アプリ（HTTP）

間違えやすいポイント

  • CGIとFastCGIの混同：CGIはリクエストごとにプロセス起動（遅い）、FastCGIは常駐（速い）。現代はFastCGIが標準。
  • リバースプロキシとフォワードプロキシの混同：リバースプロキシはサーバー側、フォワードプロキシはクライアント側に設置。
  • ProxyPassの設定忘れ：リバースプロキシを使うには適切な転送設定が必要。設定がないとアプリに到達しない。
  • localhostとポート番号の誤解：localhost（127.0.0.1）は自分自身を指す。アプリは3000番などで待ち受ける。

確認問題

Q1. CGIの問題点として正しいものはどれか。

ア）アプリケーションが常駐できない　イ）リクエストごとにプロセスを起動するため遅い　ウ）HTTPしか使えない　エ）セキュリティが弱い

答えを見る ▼

答え: イ — CGIはリクエストごとにアプリケーションを新しく起動するため、プロセス起動のオーバーヘッドが大きく遅い。

Q2. FastCGIの特徴として正しいものはどれか。

ア）リクエストごとにアプリを起動する　イ）アプリを常駐プロセスとして起動する　ウ）HTTP通信しかできない　エ）PHPでは使えない

答えを見る ▼

答え: イ — FastCGIはアプリを常駐プロセスとして起動し、Webサーバーはリクエストを渡すだけ。プロセス起動のオーバーヘッドがない。

Q3. PHPの現代的な実行方式として標準的なものはどれか。

ア）CGI　イ）mod_php　ウ）php-fpm（FastCGI）　エ）リバースプロキシ

答えを見る ▼

答え: ウ — 現代のPHP実行の標準はphp-fpm（FastCGI実装）である。CGIは過去の方式。

Q4. リバースプロキシの説明として正しいものはどれか。

ア）クライアントの通信を代理で外部に送信する　イ）Webサーバーがアプリケーションの代理としてリクエストを転送する　ウ）データベースの負荷を分散する　エ）DNS名前解決を高速化する

答えを見る ▼

答え: イ — リバースプロキシはWebサーバーがアプリケーションの代理としてリクエストを受け取り、内部のアプリケーションへ転送する仕組み。

Q5. Apacheでリバースプロキシを設定するディレクティブはどれか。

ア）ProxyPass / ProxyPassReverse　イ）FastCGI / PHP-FPM　ウ）VirtualHost / Directory　エ）ServerName / DocumentRoot

答えを見る ▼

答え: ア — Apacheでリバースプロキシを設定するにはProxyPassとProxyPassReverseディレクティブを使用する。

Q6. リバースプロキシを使用するメリットとして適切でないものはどれか。

ア）HTTPS処理をWebサーバー側で集約できる　イ）静的ファイルを高速に返せる　ウ）アプリケーションの実行速度が向上する　エ）アプリを外部に直接公開しなくてよい

答えを見る ▼

答え: ウ — リバースプロキシはアプリケーションの実行速度そのものを向上させるわけではない。通信の前段を担当する。

Q7. Node.js / PythonアプリとApacheを連携させる際に使用される方式はどれか。

ア）FastCGI　イ）mod_php　ウ）リバースプロキシ　エ）CGI

答えを見る ▼

答え: ウ — Node.jsやPythonアプリはApacheだけでは動かないため、リバースプロキシを使ってApacheからアプリに転送する。

Q8. FastCGIとリバースプロキシの違いとして正しいものはどれか。

ア）FastCGIはPHP向け、リバースプロキシはNode.js/Python向けの連携方式　イ）FastCGIはクライアント側、リバースプロキシはサーバー側の技術　ウ）両方とも同じ方式である　エ）FastCGIはHTTPしか使えない

答えを見る ▼

答え: ア — FastCGI（php-fpm）は主にPHP、リバースプロキシは主にNode.js/Python/Ruby向けの連携方式。

Q9. PythonのWebアプリケーション連携方式として標準的なものはどれか。

ア）CGI　イ）mod_python　ウ）WSGI（Gunicorn / uWSGI）　エ）Rack

答えを見る ▼

答え: ウ — Pythonの標準的な連携方式はWSGI（Web Server Gateway Interface）で、GunicornやuWSGIが実装。

Q10. リバースプロキシを使用する際、アプリケーションが待ち受けるアドレスとして適切なものはどれか。

ア）203.0.113.10:80　イ）example.com:443　ウ）127.0.0.1:3000　エ）192.168.1.10:3306

答え: ウ — アプリケーションは通常localhost（127.0.0.1）の任意のポート（3000番など）で待ち受け、Webサーバーがリバースプロキシとして転送する。
