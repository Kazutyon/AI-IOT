1. REST API / JSON

現代のWebアプリケーションは、HTMLを直接生成するのではなく、API（データ）とフロントエンド（画面）を分離して構築する。その中心となる仕組みがREST APIとJSON。

なぜREST API / JSONが必要なのか

従来のWeb（LAMP時代）はPHPがHTMLを生成、サーバー側で画面を作る、ブラウザは表示するだけという構造だった。しかし現代はSPA（React/Vue）、モバイルアプリ、外部サービス連携、非同期通信（Ajax/fetch）など、画面とデータを分離する必要がある。

REST APIとは

REST APIは、HTTPを使ってデータをやり取りするAPIの設計方式。特徴：URLが「リソース」を表す、HTTPメソッド（GET/POST/PUT/DELETE）で操作を表す、レスポンスはJSONが基本。

操作HTTPメソッドURL
商品一覧を取得GET/api/items
商品詳細を取得GET/api/items/10
商品を登録POST/api/items
商品を更新PUT/api/items/10
商品を削除DELETE/api/items/10

JSONとは

JSON（JavaScript Object Notation）は、軽量で読みやすいデータ形式。特徴：人間が読める、JavaScriptと相性が良い、ほぼすべての言語で扱える、APIの標準形式。

JSONの成功/失敗の標準形

ケースレスポンス例説明
成功時 { "status": "ok", "data": { "id": 10, "name": "Tシャツ", "price": 1980 } } status: okで成功を明示、dataに実際のデータ
失敗時 { "status": "error", "message": "無効なパラメータです" } status: errorで失敗を明示、messageにエラー内容

2. 役割分担

Webサーバーとアプリケーションは役割が明確に分かれている。この役割分担を理解しないと、Webサーバーに不要な処理をさせてしまう、アプリ側に負荷が集中する、セキュリティが弱くなる、設計が破綻するという問題が起きる。

Webサーバーの役割

  • リクエストの受付（入口）：ドメインを受け取る、HTTPS（SSL/TLS）を処理する、仮想ホストでサイトを振り分ける
  • 静的ファイルの配信：HTML、CSS、JavaScript、画像を高速に返す
  • アプリへの転送：PHP → FastCGI（php-fpm）、Node.js/Python → リバースプロキシ
  • アクセス制御・ログ・キャッシュ：access.log/error.log、IP制限、Basic認証、キャッシュ制御、リダイレクト

アプリケーションの役割

  • ビジネスロジック：ログイン処理、商品検索、注文処理、バリデーション
  • DBアクセス：SQL実行、データ取得、更新・削除（WebサーバーはDBに触れない）
  • JSONの生成（REST API）：APIのレスポンス（JSON）はアプリが生成
  • HTMLテンプレートの生成：Blade（PHP）、Jinja2（Python）、EJS（Node）
領域Webサーバー（Apache）アプリ（PHP/Python/Node）
入口ドメイン/HTTPS触れない
静的ファイル高速に返す返さない
動的処理アプリへ転送実行する
DB触れないアクセスする
ログaccess.log/error.logアプリログ
認証Basic認証ログイン処理
API入口として受けるJSONを生成

認証の役割分担（Cookie / Session / Token）

領域役割内容
ブラウザCookieの保存Session IDやTokenを保持
Webサーバー（Apache）入口Cookie/Tokenを「転送するだけ」
アプリケーション認証処理の本体Session管理/Token検証/ログイン処理

エラーハンドリングの役割分担

Webサーバーとアプリケーションは、返すべきエラーの種類が明確に分かれている。

Webサーバー（Apache）が返すエラー

  • 404 Not Found：静的ファイルが存在しない、ルーティングに到達する前にApacheが判断
  • 403 Forbidden：ディレクトリへのアクセス禁止、Basic認証の失敗、Apacheの設定による拒否
  • 413 Payload Too Large：Apacheの設定値を超えるファイルサイズ
  • 502 / 503：アプリが応答しない、FastCGI/リバースプロキシ先がダウンしている、php-fpmが停止している

アプリケーションが返すエラー（API側）

  • 400 Bad Request：パラメータ不足、バリデーションエラー、不正な形式のリクエスト
  • 401 Unauthorized：ログインが必要、Tokenが無効、Sessionが切れている
  • 404 Not Found：商品IDが存在しない、ユーザーが見つからない（※Apacheの404とは意味が異なる）
  • 500 Internal Server Error：例外発生、DBエラー、ロジックのバグ
領域返すエラー意味
Webサーバー（Apache）403 / 404 / 413 / 502 / 503入口・静的ファイル・プロキシの問題
アプリケーション400 / 401 / 404 / 500入力・認証・データ・内部処理の問題

404の違い（重要）

発生元意味
Apacheの404ファイルが存在しない、ルーティングに到達していない、URLが間違っている
APIの404データが存在しない、商品IDが見つからない、ユーザーが存在しない

間違えやすいポイント

  • Webサーバーとアプリの役割の混同：Webサーバーは「入口」、アプリは「処理」。DBアクセスはアプリの責務。
  • Cookieに認証情報を直接保存する誤り：CookieにはSession IDだけを保存し、実際のユーザー情報はSession（サーバー側）に保存する。
  • 404エラーの原因特定ミス：Apacheの404（ファイル不在）とAPIの404（データ不在）は意味が異なる。ログで確認が必要。
  • Basic認証とログイン処理の混同：Basic認証はWebサーバーが担当、フォームログインはアプリが担当。
  • 502エラーの原因：502 Bad Gatewayはアプリがダウンしている可能性。Webサーバーの問題ではない。

確認問題

Q1. REST APIの説明として正しいものはどれか。

ア）HTMLを生成するためのテンプレートエンジン　イ）HTTPを使ってデータをやり取りするAPIの設計方式　ウ）データベースのクエリ言語　エ）Webサーバーの設定ファイル

答えを見る ▼

答え: イ — REST APIはHTTPを使ってデータをやり取りするAPIの設計方式。URLがリソースを表し、HTTPメソッドで操作を表す。

Q2. JSONの特徴として適切でないものはどれか。

ア）軽量で読みやすい　イ）APIの標準形式　ウ）HTMLを記述するための言語　エ）ほとんどの言語で扱える

答えを見る ▼

答え: ウ — JSONはデータ形式であり、HTMLを記述するための言語ではない。

Q3. REST APIで商品削除を行う際に使用するHTTPメソッドはどれか。

ア）GET　イ）POST　ウ）PUT　エ）DELETE

答えを見る ▼

答え: エ — REST APIではDELETEメソッドでリソースの削除を表す。

Q4. Webサーバーの役割として適切でないものはどれか。

ア）静的ファイルの配信　イ）ビジネスロジックの実行　ウ）HTTPSの処理　エ）アクセスログの記録

答えを見る ▼

答え: イ — ビジネスロジックの実行はアプリケーションの役割。Webサーバーは入口と配信を担当する。

Q5. アプリケーションの役割として適切なものはどれか。

ア）静的ファイルの配信　イ）データベースアクセス　ウ）仮想ホストの設定　エ）アクセスログの記録

答えを見る ▼

答え: イ — DBアクセスはアプリケーションの役割。WebサーバーはDBに触れない。

Q6. 認証情報の役割分担として正しいものはどれか。

ア）Cookieの管理はアプリの役割　イ）Sessionの管理はWebサーバーの役割　ウ）認証処理の本体はアプリの役割　エ）Tokenの検証はWebサーバーの役割

答えを見る ▼

答え: ウ — 認証処理の本体（ログイン処理、Session管理、Token検証）はすべてアプリケーションの役割。

Q7. アプリケーションが応答しない場合にWebサーバーが返すエラーはどれか。

ア）400 Bad Request　イ）401 Unauthorized　ウ）404 Not Found　エ）502 Bad Gateway

答えを見る ▼

答え: エ — 502 Bad Gatewayはアプリがダウンしている、FastCGI/リバースプロキシ先が応答しない場合にWebサーバーが返す。

Q8. APIの404エラーが示す意味として正しいものはどれか。

ア）リクエストされたファイルがサーバーに存在しない　イ）リクエストされたデータが存在しない　ウ）認証エラー　エ）サーバー内部エラー

答えを見る ▼

答え: イ — APIの404は「商品IDが見つからない」「ユーザーが存在しない」など、データが存在しないことを示す。

Q9. バリデーションエラー（パラメータ不足）の場合にAPIが返すべきステータスコードはどれか。

ア）200 OK　イ）400 Bad Request　ウ）404 Not Found　エ）500 Internal Server Error

答えを見る ▼

答え: イ — バリデーションエラーやパラメータ不足はクライアント側の問題であり、400 Bad Requestを返す。

Q10. Webサーバーとアプリケーションの役割分担の目的として適切でないものはどれか。

ア）性能向上（静的ファイルはWebサーバーが高速に返す）　イ）セキュリティ強化（アプリを直接公開しない）　ウ）処理の二重化による冗長性向上　エ）保守性向上（ログやアクセス制御を集約）

答えを見る ▼

答え: ウ — 役割分担の目的は処理の二重化（冗長性）ではない。性能、セキュリティ、保守性の向上が目的。
