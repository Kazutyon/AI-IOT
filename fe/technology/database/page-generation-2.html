1. Webサーバーのレスポンスの中身

Webサーバーは、ブラウザからのリクエストに対してHTTPレスポンスを返すことでWebページを表示させる。レスポンスは次の2つで構成される：レスポンスヘッダ（HTTPの情報）、レスポンスボディ（実際の中身：HTML/JSON/CSS/JS/画像）。

レスポンスボディの種類

CSStext/cssデザイン情報。HTMLの見た目を整える

2. Content-TypeとMIMEタイプ

Content-Typeは、レスポンスボディの種類（形式）をブラウザに伝えるヘッダ。ブラウザはContent-Typeを見て、HTMLとして表示するのか、CSSとして適用するのか、JavaScriptとして実行するのか、JSONとしてデータ処理するのか、画像として描画するのかを判断する。

MIMEタイプ内容
text/htmlHTML（Webページ本体）
text/cssCSS（デザイン情報）
text/javascriptJavaScript（動的処理）
application/jsonJSON（APIの返却データ）
image/pngPNG画像
image/jpegJPEG画像
application/pdfPDF文書

日本語を扱う場合は、charset（文字コード）の指定が重要。Content-Type: text/html; charset=UTF-8のように指定する。

3. URLパラメータ（?id=10の仕組み）

URLパラメータとは、URLの末尾に「?」以降で付ける追加情報のこと。例：https://example.com/item?id=10&sort=asc

要素説明
パスどの処理かを判断する
クエリ文字列どのデータかを判断する。キー=値の形式（&で複数指定）

URLパラメータが使われる場面

  • 商品詳細ページ（/item?id=10）
  • 検索結果（/search?keyword=apple&page=2）
  • 一覧のページ切り替え（?page=3）
  • 並び替え（?sort=price）
  • フィルタリング（?category=book&limit=20）

URLパラメータの制約・注意点

  • URLに含まれるためブラウザのアドレスバーに見える
  • 機密情報（パスワード・個人情報）を載せてはいけない
  • 日本語などはURLエンコードが必要
  • 長すぎるとブラウザやサーバーの制限にかかることがある
  • GETのURLパラメータはキャッシュの対象になる

4. GETとPOST

HTTPには複数のメソッドがあるが、Webページ生成で最も重要なのはGETとPOSTの2つ。GETはデータ取得（URLにパラメータ）、POSTはデータ送信（ボディにデータ）。

項目GETPOST
データの送信先URL（クエリ文字列）HTTPボディ
アドレスバー表示表示される表示されない
ブックマークできるできない
再読み込み安全（副作用なし）再送信になる（注意）
キャッシュされるされない
主な用途ページ表示・検索フォーム送信・登録処理

なぜGETはアドレスバーに残るのか

  1. GETは「リソース取得」のためのメソッド。必要な情報はすべてURLに含めるという設計
  2. URLパラメータは「URLの一部」だからアドレスバーに表示される
  3. GETは「再読み込みしても安全」副作用がないため、URLをそのまま再利用できる
  4. ブックマーク・共有が前提。URLが見える必要がある

なぜPOSTはアドレスバーに残らないのか

  • データはURLではなくHTTPボディに入る
  • 副作用があるため再読み込みが危険
  • ブックマークや共有に向かない
  • 履歴に残すべきではない

GET / POST の使い分け

  • ページを開く → GET
  • データを送信する → POST

この原則を守ることで、URLが整理される、セキュリティ事故を防げる、ブラウザの動作（キャッシュ・履歴）が正しく働く。

間違えやすいポイント

  • GETでパスワードを送信してはいけない：GETはURLにパラメータが表示されるため、パスワードなどの機密情報を送信するのは絶対にNG。
  • Content-Typeの指定漏れ：Content-Typeが正しく設定されていないと、ブラウザが正しく表示できない。
  • URLパラメータのエンコード忘れ：日本語や特殊文字を含む場合はURLエンコードが必要。
  • POSTの再読み込み問題：POSTページで再読み込みすると「再送信」の確認ダイアログが表示される。重要な処理ではリダイレクト（Post/Redirect/Getパターン）を使う。
  • MIMEタイプの誤り：JSONを返すAPIでtext/htmlを返すと、フロントエンドが正しく処理できない。

確認問題

Q1. HTTPレスポンスの構成要素として正しい組み合わせはどれか。

ア）リクエストライン / ヘッダ / ボディ　イ）ステータスライン / ヘッダ / ボディ　ウ）メソッド / URL / HTTPバージョン　エ）ステータスコード / メソッド / ボディ

答えを見る ▼

答え: イ — HTTPレスポンスは「ステータスライン」「ヘッダ」「ボディ」で構成される。リクエストラインはリクエストの構成要素。

Q2. JSONデータのMIMEタイプはどれか。

ア）text/html　イ）text/css　ウ）application/json　エ）image/png

答えを見る ▼

答え: ウ — JSONのMIMEタイプはapplication/json。APIのレスポンスでよく使用される。

Q3. URLパラメータ（クエリ文字列）の説明として正しいものはどれか。

ア）HTTPボディに格納されるデータ　イ）URLの末尾に「?」以降で付ける追加情報　ウ）HTTPヘッダに含まれる情報　エ）Cookieに保存される情報

答えを見る ▼

答え: イ — URLパラメータはURLの末尾に「?」以降で付けるキー=値形式の追加情報である。

Q4. GETメソッドの特徴として適切でないものはどれか。

ア）データ取得に使用する　イ）URLにパラメータを付ける　ウ）パスワードなどの機密情報を送信するのに適している　エ）ブックマーク可能

答えを見る ▼

答え: ウ — GETはURLにパラメータが表示されるため、パスワードなどの機密情報を送信するのには適していない。そのような場合はPOSTを使用する。

Q5. POSTメソッドの特徴として適切なものはどれか。

ア）データはURLのクエリ文字列に格納される　イ）アドレスバーにパラメータが表示される　ウ）ブックマーク可能　エ）ログインや登録処理に使用される

答えを見る ▼

答え: エ — POSTはデータ送信に使用され、ログインや会員登録などの処理に適している。

Q6. CSSファイルのMIMEタイプはどれか。

ア）text/html　イ）text/css　ウ）text/javascript　エ）application/json

答えを見る ▼

答え: イ — CSSファイルのMIMEタイプはtext/css。ブラウザはこれを見てスタイルシートとして適用する。

Q7. URLパラメータの制約として正しいものはどれか。

ア）パスワードなどの機密情報を載せてもよい　イ）日本語はそのまま使える　ウ）長すぎるとブラウザやサーバーの制限にかかることがある　エ）キャッシュの対象にならない

答えを見る ▼

答え: ウ — URLパラメータが長すぎると、ブラウザやWebサーバーの制限（約2000〜8000文字）にかかることがある。

Q8. 次のうち、GETメソッドが適切に使われている例はどれか。

ア）ログインフォームの送信　イ）会員登録フォームの送信　ウ）商品検索フォームの送信　エ）クレジットカード情報の送信

答えを見る ▼

答え: ウ — 商品検索はデータ取得が目的であり、検索キーワードをURLパラメータとして送信するGETが適切。

Q9. 日本語を含むURLパラメータを扱う際に必要な処理はどれか。

ア）Base64エンコード　イ）URLエンコード　ウ）暗号化　エ）圧縮

答えを見る ▼

答え: イ — 日本語や特殊文字を含むURLパラメータは、URLエンコード（パーセントエンコーディング）が必要。

Q10. POSTメソッドでデータを送信する際、データが格納される場所はどれか。

ア）URLのクエリ文字列　イ）HTTPヘッダ　ウ）HTTPボディ　エ）ステータスライン

答えを見る ▼

答え: ウ — POSTメソッドでは、送信データはHTTPボディに格納される。GETメソッドではURLのクエリ文字列に格納される。
