1. 仮想ホスト

仮想ホスト（Virtual Host / Server Block）とは、1台のサーバーで複数のWebサイトを公開するための仕組み。例：example.com、shop.example.com、test.example.comを1つのApache/Nginxで同時に公開できる。

なぜ仮想ホストが必要なのか

Webサーバーは通常、1つのIPアドレスで動作する。しかし実際の運用では：1台のサーバーで複数のサイトを公開したい、サイトごとに公開ディレクトリを分けたい、サイトごとにログを分けたい、サイトごとにSSL証明書を設定したい、サイトごとに設定（エラーページ・リダイレクト）を変えたい。これを実現する仕組みが仮想ホスト。

仮想ホストの構造（3階層）

  1. サーバー全体の設定（共通設定）：ポート番号（80/443）、モジュール読み込み、ログ形式、性能設定（gzip/Keep-Alive）
  2. 仮想ホスト（ドメイン単位の設定）：ServerName/server_name（どのドメインか）、DocumentRoot/root（公開ディレクトリ）、サイトごとのログ、サイトごとのエラーページ、HTTPS設定（証明書）
  3. ロケーション（URLパス単位の設定）：/img/は静的ファイル、/api/はアプリケーションへ渡す、/admin/はアクセス制限

仮想ホストの仕組み

ブラウザはリクエスト時にHostヘッダを送る。例：GET / HTTP/1.1 Host: example.com。WebサーバーはHostヘッダを確認し、example.comに一致する仮想ホストを探し、その仮想ホストのDocumentRootを使ってレスポンスを返す。これにより、同じIPアドレスでも複数サイトを公開できる。

項目Apache（VirtualHost）Nginx（serverブロック） 仮想ホスト定義那样<VirtualHost *:80>server { listen 80; } ドメイン指定ServerName example.comserver_name example.com; 公開ディレクトリDocumentRoot /var/www/exampleroot /var/www/example; 設定の上書き那样.htaccess可能那样不可（一括管理）

2. ファイル配置（DocumentRoot / public_html）

Webサーバー（Apache/Nginx）は、「どのディレクトリをWeb公開するか」を設定する必要がある。その公開ディレクトリをDocumentRoot（ドキュメントルート）と呼ぶ。

DocumentRootの役割

DocumentRootは、URLの「/」に対応するフォルダ。URL → ファイルパスの変換の起点になる。

URL実際のファイルパス（DocumentRoot=/var/www/example）
//var/www/example/index.html
/img/a.png/var/www/example/img/a.png
/css/style.css/var/www/example/css/style.css

public_htmlとは

サーバーによっては、ユーザーごとにWeb公開ディレクトリを持つ仕組みがある。例：/home/hiro/public_html。URLは：http://example.com/~hiro/。共有サーバーでユーザーごとに公開領域を分ける。ApacheのUserDir機能で実現される。Nginxでは一般的ではない。

項目DocumentRootpublic_html
単位サイト単位ユーザー単位
設定場所仮想ホストUserDir（Apache）
URL/ から始まる/~ユーザー名/
用途通常のWebサイト公開共有サーバーでの個人公開

ファイル配置の基本ルール

WebサーバーはDocumentRoot以下のファイルだけを公開する。DocumentRootの外のファイルは公開されない。セキュリティ上、公開領域と非公開領域を分ける。アプリケーションのソースコードはDocumentRootの外に置くのが基本。

3. カスタムエラーページ（404 / 500）

Webサーバーは、リクエストされたファイルが存在しない場合（404）やアプリケーションがエラーを返した場合（500）に、エラーページを返す。このエラーページは、Webサーバー側で自由に差し替えることができる。これをカスタムエラーページと呼ぶ。

なぜカスタムエラーページが必要なのか

デフォルトのエラーページは、デザインが統一されていない、サイトの世界観と合わない、ユーザーが次に何をすべきかわからない、セキュリティ上内部情報が表示される場合がある。

エラーページのポイント内容
404ページ「ページが見つかりません」、トップページへのリンク、検索フォーム、サイトマップへの導線
500ページ「サーバーエラーが発生しました」、時間を置いて再度アクセスする案内、問い合わせ先の提示（必要に応じて）

4. ログ（access.log / error.log）

Webサーバー（Apache/Nginx）は、すべてのアクセスとエラーをログとして記録する。ログは、障害調査、性能分析、不正アクセス検知、運用改善に不可欠な情報源である。

なぜログが必要なのか

Webサイトを運用する側は、サーバー内部で何が起きているかをブラウザから直接知ることができない。そのため、サーバー側では次の情報をログとして記録しておく必要がある：どのURLにアクセスされたか、どのIPからアクセスされたか、どのステータスコードが返されたか、どのページでエラーが発生したか、処理に何秒かかったか、どのブラウザ/端末からのアクセスか。

access.log（アクセスログ）

access.logは、すべてのリクエストを記録するログ。記録される主な情報：アクセス元IPアドレス、アクセスされたURL、ステータスコード（200/404/500など）、レスポンスサイズ、レスポンス時間、User-Agent（ブラウザ情報）、リファラ（どこから来たか）。

error.log（エラーログ）

error.logは、Webサーバー内部で発生したエラーを記録するログ。記録される主な情報：ファイルが存在しない（404）、権限エラー（403）、アプリケーションのエラー（500）、設定ミス、サーバー内部の警告・エラー。

ApacheとNginxのログ設定

Webサーバー設定場所設定例
ApacheVirtualHost内ErrorLog "error.log"
CustomLog "access.log" combined
Nginxserverブロック内access_log /var/log/nginx/access.log main;
error_log /var/log/nginx/error.log;

どちらも仮想ホスト（server）単位でログを分けられる。

間違えやすいポイント

  • 仮想ホストと物理サーバーの混同：仮想ホストは「1台の物理サーバーで複数サイト」の仕組み。仮想マシンとは別の概念。
  • DocumentRootとpublic_htmlの混同：DocumentRootはサイト単位、public_htmlはユーザー単位の公開ディレクトリ。
  • 404エラーページのカスタマイズ忘れ：デフォルトの404ページはユーザーフレンドリーではない。カスタマイズが望ましい。
  • ログの確認不足：トラブル発生時にまずログを確認する習慣がないと、原因特定に時間がかかる。
  • access.logとerror.logの混同：アクセス記録はaccess.log、エラー記録はerror.log。役割が異なる。

確認問題

Q1. 1台のサーバーで複数のWebサイトを公開する仕組みを何と呼ぶか。

ア）ロードバランシング　イ）仮想ホスト　ウ）リバースプロキシ　エ）キャッシュ

答えを見る ▼

答え: イ — 仮想ホスト（Virtual Host / Server Block）は、1台のサーバーで複数のWebサイトを公開する仕組み。

Q2. 仮想ホストの振り分けに使用されるHTTPヘッダはどれか。

ア）User-Agent　イ）Accept　ウ）Host　エ）Content-Type

答えを見る ▼

答え: ウ — ブラウザはリクエスト時にHostヘッダを送信し、Webサーバーはこの値を見て仮想ホストを振り分ける。

Q3. Webサーバーが公開するファイルを探す起点ディレクトリはどれか。

ア）public_html　イ）DocumentRoot　ウ）ServerRoot　エ）LogRoot

答えを見る ▼

答え: イ — DocumentRootはWebサーバーが公開するファイルを探す起点ディレクトリ。URLの「/」に対応する。

Q4. レンタルサーバーでユーザーごとに与えられる公開ディレクトリはどれか。

ア）/etc/public　イ）/var/www　ウ）public_html　エ）DocumentRoot

答えを見る ▼

答え: ウ — public_htmlは共有サーバーでユーザーごとに与えられる公開ディレクトリ。URLは/~ユーザー名/でアクセスする。

Q5. カスタムエラーページを設定する主な理由として適切でないものはどれか。

ア）デザインを統一するため　イ）ユーザーに次に何をすべきか示すため　ウ）サーバーの内部情報を隠すため　エ）レスポンス速度を上げるため

答えを見る ▼

答え: エ — カスタムエラーページはレスポンス速度向上が目的ではない。ユーザー体験向上と情報漏洩防止が主な目的。

Q6. Webサーバーのアクセス記録を保存するログファイルはどれか。

ア）error.log　イ）access.log　ウ）system.log　エ）debug.log

答えを見る ▼

答え: イ — access.logはすべてのリクエストを記録するアクセスログ。error.logはエラーを記録する。

Q7. access.logに記録されない情報はどれか。

ア）アクセス元IPアドレス　イ）リクエストされたURL　ウ）データベースのクエリ内容　エ）ステータスコード

答えを見る ▼

答え: ウ — データベースのクエリ内容はaccess.logには記録されない。アプリケーション側のログに記録される。

Q8. error.logに記録される情報として適切なものはどれか。

ア）アクセス元のブラウザ情報　イ）404 Not Foundエラー　ウ）レスポンスサイズ　エ）リファラ情報

答えを見る ▼

答え: イ — error.logにはファイルが存在しない（404）、権限エラー（403）、アプリケーションエラー（500）などが記録される。

Q9. ログの「リファラ」が示す情報はどれか。

ア）アクセス元のIPアドレス　イ）どのページから遷移してきたか　ウ）使用しているブラウザ　エ）リクエストの処理時間

答えを見る ▼

答え: イ — リファラは、ユーザーがどのページから遷移してきたかを示す情報。広告効果測定などに使われる。

Q10. 仮想ホストの階層構造として正しいものはどれか。

ア）ロケーション → 仮想ホスト → サーバー全体　イ）サーバー全体 → 仮想ホスト → ロケーション　ウ）仮想ホスト → サーバー全体 → ロケーション　エ）ロケーション → サーバー全体 → 仮想ホスト

答えを見る ▼

答え: イ — 正しい階層は「サーバー全体の設定 → 仮想ホスト（ドメイン単位） → ロケーション（URLパス単位）」の順。
