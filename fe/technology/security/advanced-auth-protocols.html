1. 所持要素認証の仕組みと弱点

所持要素認証は、本人が「持っている物」（スマホ、認証アプリ、ハードウェアキー）を使って本人確認を行う方式です。内部では、サーバーとデバイスが共有する秘密鍵と、時刻や乱数を用いた暗号計算（TOTPやチャレンジレスポンス）によって動作しています。

代表的なTOTP（認証アプリ）は、QRコードで共有した秘密鍵と現在時刻（30秒区切り）をHMAC-SHA1で計算し、6桁の数字を生成します。一方、プッシュ通知型は、サーバーが生成した乱数（チャレンジ）に対し、端末内の秘密鍵で署名（レスポンス）を返すことで本人確認を行います。

一方で、SMS認証は技術的に弱点が多いとされています。SMSは通信規格（SS7）の脆弱性により盗聴されるリスクがあり、何より「SIMスワップ攻撃」により電話番号自体を乗っ取られると、認証コードが攻撃者の端末に届いてしまうためです。そのため、実務では認証アプリやFIDO2セキュリティキーの利用が強く推奨されています。

2. パスワードレス認証（FIDO2 / パスキー）

FIDO2（WebAuthn）やパスキーは、パスワードを完全に排除し、公開鍵暗号を利用して認証を行う次世代の方式です。端末内の「セキュア領域（Secure Enclave / TPM）」で秘密鍵を生成・保存し、公開鍵のみをサーバーに登録します。ログイン時は、サーバーからの乱数（チャレンジ）に対し、秘密鍵で署名して応答します。

この方式において、指紋や顔認証などの生体認証は「本人確認そのもの」ではなく、「セキュア領域に保存された秘密鍵の利用を許可する（ロックを解除する）」ための手段として機能します。生体情報そのものがサーバーに送信されることは決してありません。

FIDO2が極めて高い「フィッシング耐性」を持つ理由は、オリジン（ドメイン）チェックを行う仕組みがあるためです。秘密鍵は「example.com」用として生成されるため、攻撃者が「examp1e.com」のような偽サイトを用意しても、ブラウザがドメインの不一致を検知し、署名が成立しないようブロックします。

3. 認証プロトコル（OAuth2 / OIDC）

現代のWebサービスでは、複数のサービス間で安全に連携するための標準プロトコルとしてOAuth2とOIDCが利用されています。OAuth2は「認可（Authorization）」を扱い、ユーザーのパスワードを共有せずに、サードパーティのアプリに限定されたアクセス権限（スコープ）を委譲する仕組みです。

一方、OIDC（OpenID Connect）はOAuth2を拡張し、「認証（Authentication）」機能を追加したプロトコルです。OIDCでは、認可サーバーが「IDトークン（JWT形式）」を発行し、これに含まれる署名付きのユーザー情報（名前、メールアドレス等）によって、利用者が確かに本人であることを保証します。

特にモバイルアプリやSPA（シングルページアプリケーション）では、認可コードが盗聴されるリスクがあるため、「PKCE（Proof Key for Code Exchange）」の使用が必須となっています。PKCEは、クライアント側で生成したランダムな文字列（code_verifier）のハッシュ値を事前に送信し、トークン交換時に元の文字列を提示させることで、コードが盗まれても悪用できないようにする仕組みです。

確認問題

Q1. 所持要素認証において、現在時刻と共有秘密鍵を用いて6桁の数字を生成する方式はどれか。

ア）SMS認証 イ）TOTP（Time-based One-Time Password） ウ）生体認証 エ）パスワードリスト攻撃

答えを見る ▼

答え: イ — TOTPは、サーバーと認証アプリが共有する秘密鍵と、30秒ごとに区切られた現在時刻を元にHMAC計算を行い、使い捨てパスワードを生成します。

Q2. SMS認証が技術的に弱いとされる最大の理由はどれか。

ア）通信速度が遅いためタイムアウトしやすい イ）SIMスワップ攻撃により電話番号自体を乗っ取られるリスクがある ウ）暗号化が強すぎて端末のバッテリーを消費する エ）ユーザーがコードを覚えられないため

答えを見る ▼

答え: イ — 攻撃者が携帯電話会社を騙して被害者の電話番号を自分のSIMカードに移行させる「SIMスワップ攻撃」が成立すると、SMS認証は突破されてしまいます。

Q3. FIDO2（パスキー）において、指紋や顔認証などの生体認証が担う役割はどれか。

ア）生体情報をサーバーに送信して本人確認する イ）セキュア領域に保存された秘密鍵の利用を許可する（ロック解除する） ウ）パスワードの代わりとなる複雑な文字列を生成する エ）ネットワーク通信を暗号化する鍵を配布する

答えを見る ▼

答え: イ — FIDO2では、生体認証は「秘密鍵を使ってもよいかの許可」を得るためだけに使われ、実際の本人確認は公開鍵暗号の仕組み（チャレンジレスポンス）で行われます。

Q4. FIDO2認証がフィッシング攻撃に対して極めて強い耐性を持つ技術的理由はどれか。

ア）パスワードを使わないため総当たり攻撃が不可能だから イ）オリジン（ドメイン）チェックを行い、偽サイトでは署名が成立しない仕組みになっているから ウ）生体情報は変更できないため、漏洩のリスクがゼロだから エ）通信経路が常にVPNで保護されているから

答えを見る ▼

答え: イ — FIDO2の秘密鍵は特定のドメイン（例: example.com）に紐付いて生成されるため、攻撃者が類似ドメイン（examp1e.com）の偽サイトを用意しても署名が成立せず、突破できません。

Q5. OAuth2プロトコルが主に扱うセキュリティ機能はどれか。

ア）認証（Authentication） イ）認可（Authorization） ウ）暗号化（Encryption） エ）監査（Auditing）

答えを見る ▼

答え: イ — OAuth2は「認可」を扱うプロトコルであり、ユーザーのパスワードを共有せずに、サードパーティのアプリに限定されたアクセス権限（スコープ）を委譲します。

Q6. OIDC（OpenID Connect）において、ユーザーの本人確認情報を提供するために発行されるトークンはどれか。

ア）アクセストークン イ）リフレッシュトークン ウ）IDトークン（JWT） エ）CSRFトークン

答えを見る ▼

答え: ウ — OIDCはOAuth2に認証機能を追加したもので、認証の結果としてユーザー情報を含む「IDトークン（JWT形式）」を発行します。

Q7. モバイルアプリなどで認可コードが盗聴される攻撃（コード横取り）を防ぐために必須とされる拡張機能はどれか。

ア）PKCE（Proof Key for Code Exchange） イ）HSTS（HTTP Strict Transport Security） ウ）CORS（Cross-Origin Resource Sharing） エ）SAML（Security Assertion Markup Language）

答えを見る ▼

答え: ア — PKCEは、クライアント側で生成したランダムな文字列（code_verifier）のハッシュ値を事前に送信し、トークン交換時に元の文字列を提示させることで、盗まれたコードの悪用を防ぎます。

Q8. JWT（JSON Web Token）が改ざんされていないことを保証する仕組みはどれか。

ア）Base64エンコードされているため イ）発行者の秘密鍵によるデジタル署名（Signature）が含まれているため ウ）トークンの有効期限（exp）が設定されているため エ）HTTPS通信のみで送信されるため

答えを見る ▼

答え: イ — JWTはHeader, Payload, Signatureの3部構成であり、発行者の秘密鍵で生成された署名を検証することで、内容が改ざんされていないことを保証します。

Q9. パスキー（Passkey）をクラウド経由で複数デバイス間で同期する際、セキュリティを維持するための仕組みはどれか。

ア）秘密鍵そのものを平文でクラウドサーバーに保存する イ）秘密鍵は端末ごとに生成し、クラウドには暗号化されたバックアップのみを保存し、復元には生体認証を要求する ウ）同期機能はセキュリティリスクが高いため、パスキーでは一切サポートされない エ）同期時には必ず管理者が手動で承認する必要がある

答えを見る ▼

答え: イ — パスキーのクラウド同期では、秘密鍵そのものはクラウドに保存されず、エンドツーエンドで暗号化されたバックアップデータのみが保存され、復元時にはユーザーの生体認証が必須となります。

Q10. OAuth2/OIDCを利用する際、アプリケーションが必要以上の権限を持たないようにするために適用すべき原則はどれか。

ア）最小権限の原則（Least Privilege） イ）フェイルセーフの原則 ウ）防御の多重化 エ）ゼロトラストの原則

答えを見る ▼

答え: ア — 最小権限の原則により、アプリケーションが必要とするスコープ（権限）のみを付与することで、万一トークンが漏洩した場合の被害範囲を限定できます。
