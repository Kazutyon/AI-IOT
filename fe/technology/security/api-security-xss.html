1. APIセキュリティの基礎（OAuth2 / OIDC）

現代のアプリケーションは、Web、モバイル、外部サービスなど多くのクライアントがAPIを通じてデータをやり取りします。APIは外部に公開されることが多く、攻撃者に直接触れられるため、適切な認証・認可が必須となります。その中心となるのがOAuth2とOpenID Connect（OIDC）です。

OAuth2は「認可（Authorization）」を扱うプロトコルであり、ユーザーのパスワードをサービス側に渡すことなく、限定されたアクセス権限（Scope）をサードパーティに委譲します。一方、OIDCはOAuth2を拡張し、「認証（Authentication）」機能を追加したプロトコルです。OIDCでは、認証の結果としてユーザー情報を含む「IDトークン（JWT形式）」が発行され、利用者が確かに本人であることを保証します。

これらのプロトコルで発行されるJWT（JSON Web Token）は、Header（署名方式）、Payload（ユーザー情報・有効期限）、Signature（改ざん防止の署名）の3部構成で成り立っています。サーバーは公開鍵で署名を検証することで、トークンが改ざんされていないことを確認し、安全にアクセスを許可します。

2. XSS（クロスサイトスクリプティング）の仕組みと対策

XSSは、攻撃者が悪意のあるスクリプト（JavaScriptなど）をアプリケーションに埋め込み、他のユーザーのブラウザで実行させる攻撃です。成立すると、セッションIDの盗難、Cookieの窃取、偽画面の表示によるフィッシングなど、ユーザーに成りすました悪意ある操作が可能になります。

XSSは主に3種類に分類されます。「反射型XSS」はURLパラメータなどに埋め込んだスクリプトがそのままレスポンスに反映されるもの、「永続型XSS」は掲示板やコメント欄などデータベースに保存され、他のユーザーの画面で実行されるもの、「DOMベースXSS」はJavaScriptがDOMを操作する際に、不正な入力をそのままHTMLに挿入してしまうものです。

XSS対策の核心は「出力エスケープ」です。データをHTMLに表示する際、`<` を `<`、`"` を `"` などに無害化することで、ブラウザがスクリプトとして解釈するのを防ぎます。さらに、CSP（Content Security Policy）を導入し、許可されたドメインからのみスクリプトの実行を許可することで、万が一スクリプトが埋め込まれても実行を防ぐ多層防御が現代の標準です。

確認問題

Q1. ユーザーのパスワードを共有せずに、サードパーティのアプリケーションに限定されたアクセス権限を委譲するプロトコルはどれか。

ア）SAML イ）OAuth2 ウ）OIDC エ）LDAP

答えを見る ▼

答え: イ — OAuth2は「認可（Authorization）」を扱うプロトコルであり、アクセストークンを使用して権限を委譲します。

Q2. OAuth2を拡張し、「認証（本人確認）」機能を追加し、IDトークン（JWT）を発行するプロトコルはどれか。

ア）OpenID Connect (OIDC) イ）RADIUS ウ）Kerberos エ）SFTP

答えを見る ▼

答え: ア — OIDCはOAuth2に認証機能を追加したプロトコルであり、現代のWeb認証の標準となっています。

Q3. JWT（JSON Web Token）が改ざんされていないことを保証する仕組みはどれか。

ア）Base64エンコードされているため イ）発行者の秘密鍵によるデジタル署名（Signature）が含まれているため ウ）トークンの有効期限（exp）が設定されているため エ）HTTPS通信のみで送信されるため

答えを見る ▼

答え: イ — JWTはHeader, Payload, Signatureの3部構成であり、発行者の秘密鍵で生成された署名を検証することで、内容が改ざんされていないことを保証します。

Q4. APIへのリクエストを一元管理し、認証・認可・レート制限などを適用してAPI全体を保護するコンポーネントはどれか。

ア）ロードバランサー イ）API Gateway ウ）CDN エ）DNSサーバー

答えを見る ▼

答え: イ — API Gatewayは、APIセキュリティの最前線でトークン検証やレート制限などを適用し、バックエンドのAPIサーバーを保護します。

Q5. 一定時間あたりのAPIリクエスト回数を制限し、DoS攻撃やパスワード総当たり攻撃を防ぐ仕組みはどれか。

ア）ロードバランシング イ）レート制限（Rate Limiting） ウ）キャッシング エ）圧縮

答えを見る ▼

答え: イ — レート制限は、APIの可用性を保ち、悪意のある大量アクセスからサービスを守るために必須の対策です。

Q6. 攻撃者が悪意のあるスクリプトをWebページに埋め込み、他のユーザーのブラウザで実行させる攻撃はどれか。

ア）SQLインジェクション イ）クロスサイトスクリプティング（XSS） ウ）クロスサイトリクエストフォージェリ（CSRF） エ）ディレクトリトラバーサル

答えを見る ▼

答え: イ — XSSは、ユーザーの入力がスクリプトとして実行されることで発生し、Cookieの窃取やなりすまし操作の原因となります。

Q7. XSSの種類の一つで、攻撃者の入力がデータベースに保存され、他のユーザーがそのページを表示した際にスクリプトが実行されるものはどれか。

ア）反射型XSS イ）永続型XSS ウ）DOMベースXSS エ）ブラインドXSS

答えを見る ▼

答え: イ — 永続型XSS（Stored XSS）は、掲示板やコメント欄などに入力された悪意あるコードがサーバーに保存され、不特定多数のユーザーに影響を与えるため特に危険です。

Q8. XSS対策として、データをHTMLに表示する際に特殊文字を無害な文字列（例: <）に変換する処理はどれか。

ア）ハッシュ化 イ）出力エスケープ ウ）暗号化 エ）圧縮

答えを見る ▼

答え: イ — 出力エスケープは、ブラウザが入力をHTMLタグとして解釈するのを防ぎ、XSSを未然に防ぐ最も基本的で重要な対策です。

Q9. JavaScriptがDOMを操作する際に、ユーザー入力をそのままHTMLに挿入してしまうことで発生するXSSの種類はどれか。

ア）反射型XSS イ）永続型XSS ウ）DOMベースXSS エ）サーバーサイドXSS

答えを見る ▼

答え: ウ — DOMベースXSSは、サーバーへのリクエストを伴わずにクライアント側のJavaScript処理（例: innerHTMLへの直接代入）だけで発生するため、サーバー側のWAF等では検知が難しい場合があります。

Q10. ブラウザに対して、許可されたドメインからのみスクリプトを実行するよう指示し、XSSによる被害を大幅に軽減する仕組みはどれか。

ア）HSTS イ）CSP（Content Security Policy） ウ）CORS エ）SRI

答えを見る ▼

答え: イ — CSPは、インラインスクリプトの実行禁止や信頼できるドメインのホワイトリスト設定により、万が一スクリプトが埋め込まれても実行を防ぐ強力な防御層となります。
